# 四界面切换系统 · 重构计划

> 目标：把现有透明无边框挂机游戏，重构成「四个对等主 tab（家园 / 种植 / 换装 / 工坊）+ 仓库（全局浮层）」的结构，并给仓库做成跨越所有场景的统一库存；底部 TabBar 以「招牌晃动」动画交互。
> 注：家园是第 4 个对等主 tab（非底层常驻层）；仓库是全局浮层（任意屏可打开，不计入 tab 切换位）。

---

## 0. 已确认的设计决策（共识基线）

| 项 | 结论 |
|----|------|
| 主 tab（四个对等） | **家园 / 种植 / 换装 / 工坊**（权重一致，互斥切换；家园是其中之一，非底层常驻层） |
| 仓库定位 | **全局浮层**（非 tab）：任意屏激活时都可打开，挂在底部招牌栏分隔位，绘制在 TabBar 之下使招牌始终可点 |
| 服装并入 inventory | **已确认并入**：所有权统一进 `inventory[CLOTHING]`；穿搭由 `GameManager.equipped`（slot→item_id）单独管理，不进 inventory |
| 仓库定位（旧） | ~~第 4 个 tab（统一库存面板）~~ → 已改为**全局浮层**（见上「主 tab / 仓库定位」行） |
| 仓库跨场景方式 | 靠**数据层统一**实现：三个活动只写 `GameManager.inventory`，仓库只读聚合 |
| TabBar 位置 | **底部**（横栏挂招牌，隐喻契合 + 小窗空间更干净） |
| 招牌动画 | 点 tab 时该 tab 绕顶部枢轴 Elastic 摆动（-12°→0°，过冲回弹） |
| 现有浮窗 | 灵感浮窗、电话双击→订单中心/商城 保持原样叠加，不并入 tab |
| 窗口形态 | 右下角、透明、无边框（已有，确认 `project.godot` 设置） |
| 灵感阈值 | **累计获得**（`inspiration_total_earned` 单调递增），避免花掉后蓝图又锁 |
| 种植玩法 | 商店买花盆(PLACEABLE)+种子(SEED) → 种植屏 **6 槽**(暂定)放盆种入 → 苗/成长/成熟三阶段(时长可配,墙钟挂机) → 成熟采摘进 CROP；商店扩容花盆槽位**后续再定** |
| 桌面装饰花盆 | 限量几盆经现有 PlacementManager 摆桌面作装饰（与种植屏 6 功能槽分离），上限**后续再定** |
| 制作玩法 | 工坊：灵感达阈值解锁蓝图 → 蓝图列所需材料 → 用材料制作产出服装/制作物；植物分解得染料+植物材料 |
| ItemData 分类 | `CLOTHING / SEED / CROP / MATERIAL / BLUEPRINT / CRAFT / PLACEABLE` |
| 分解功能 | 工坊「分解」支持**所有带 `decompose_recipe` 的物品**（不限于 CROP），产物进 inventory[MATERIAL] |
| 家园·服装展架 | 桌面可拖动摆件；点击开上架面板；6 槽(暂定)可重复同款 CLOTHING；每槽显示图标+**库存数量=拥有总数−穿戴数，自动补充**；顾客随机选有货槽购买后消失(不再无条件给金币)；售价取 `ItemData.price` |

> 已确认项见上表；仅余少量开放项见文末「待确认」（执行节奏已定为**逐步执行、每步由用户确认**）。

## 阶段 0 · 数据层统一（所有屏幕的前置依赖）

> 仓库要「跨越所有场景」，必须先有统一的数据底座。这一阶段不碰 UI。

- **步骤 0.1 — 新建 `ItemData` 资源类** `scripts/resources/item_data.gd`
  - `class_name ItemData`
  - 字段：`id:String`、`display_name:String`、`icon:Texture2D`、`category`（枚举 `CLOTHING / SEED / CROP / MATERIAL / BLUEPRINT / CRAFT / PLACEABLE`）、`description:String`
  - `SEED`(种子)/`CROP`(作物)/`MATERIAL`(材料·含染料与植物材料)/`BLUEPRINT`(蓝图)/`CRAFT`(制作物·除服装外的成品) 为本次新增分类
  - 可选：`world_texture:Texture2D`、`placeable_scene:PackedScene`（摆放物用，沿用 ProductData 已有字段）
- **步骤 0.2 — `GameManager` 增加统一库存**
  - 新增 `inventory: Dictionary`（key=`id`，value=`count: int`；物品元数据不存这里，改由 `_item_registry` 按 id 运行时解析）
  - 新增 `_item_registry: Dictionary`（id→ItemData，运行时注册，非存档）
  - 方法：`register_item(data)`、`get_item(id)`、`add_item(id, n=1)`、`remove_item(id, n=1)`、`has_item(id)`、`get_count(id)`、`get_by_category(cat)`、`get_total_count()`
  - 信号：`inventory_changed`（仓库角标 / 网格监听）
  - 存档：`user://inventory.cfg`（仅存 count）；首次启动若无 inventory.cfg 但有旧 `warehouse.cfg` 则一次性迁移（id→count=1）
  - 设计说明：inventory 只存 count，元数据走注册表，避免 autoload(GameManager) 在 main 注册物品前就需要实例数据（载入顺序更稳）
- **步骤 0.3 — 接入现有 phone 购买 / 收货（PLACEABLE）**
  - `main._ready` 在 `_ensure_product_pool()` 后把 `product_pool` 逐个 `GameManager.register_item(p)`（让元数据进注册表）
  - 收货结算点改写：`confirm_receipt` / `confirm_receipt_one` → `_unlock_internal(id)` → 现改为 `GameManager.add_item(id, 1)`，写入 `inventory`（category 由注册表 ItemData 决定）
  - 保留 `has_product(id)` / `get_owned_ids()` 作为委托给 inventory 的向后兼容包装（旧仓库面板暂不受影响）；`unlock_product` 同样委托 `add_item`
  - 注：购买当下只生成订单（扣金币），真正入库发生在「到货确认收货」时，与原有流程一致
- **步骤 0.4 — 接入现有衣橱「拥有」（CLOTHING）【✅ 已实现】**
  - `ClothesData` 改 `extends ItemData`（`category=CLOTHING`），删除与基类重复的 `id/display_name/price`；保留 `texture/icon_texture/slot/inspiration_value`；`_init` 设默认 `price=50`（.tres 显式值覆盖）。现有 `.tres` 无需改动
  - 穿搭状态搬到 `GameManager.equipped: Dictionary`（slot→item_id），**不进 inventory**；`inventory` 只管「拥有」。新增 `equip/unequip/get_equipped/get_worn_count/is_worn` + `equipped_changed` 信号 + 存档 `user://equipped.cfg`；一次性迁移旧 `user://wardrobe.cfg`（slot→resource_path 转 slot→id，保住当前穿搭）
  - 所有权：`main._ready` 注册 `clothes_pool` 进注册表 + `GameManager.seed_starter_clothing()` **一次性播种**（每款各 1 件作初始衣橱，`_clothing_seeded` flag 存 `inventory.cfg [meta]`，防止售出/分解后又刷回来）
  - 衣橱 `wardrobe.gd`：背包改从 `get_by_category(CLOTHING)` 读「已拥有」（经注册表解析回 ClothesData 取贴图）；穿脱走 `GameManager.equip/unequip`；`_load_outfit` 从 `GameManager.equipped` 还原；删除本地 `_equipped` 与 `wardrobe.cfg` 存取
  - 接受条件：换装仍能穿脱 + 存档；服装数据已进 `inventory[CLOTHING]`（仓库「服装」分类的**可视化**待阶段 2.2 仓库屏落地）；被穿服装可经 `get_worn_count` 从展架库存扣除

- **步骤 0.5 — 新建 `SeedData` 资源类** `scripts/resources/seed_data.gd`（extends ItemData，category=SEED）【✅ 已实现】
  - 字段：`sprout_minutes`、`growing_minutes`、`mature_minutes`（苗/成长/成熟三阶段时长，检查器可配）、`crop_output_id`（成熟产出的 CROP 物品 id）、`stage_icons`（可选：三阶段显示图标）
  - 例：玫瑰种子 sprout=2 / growing=5 / mature=3 分钟 → 产出「玫瑰(CROP)」
  - 已落地：类含 `total_grow_minutes()` 与 `stage_at(grown_minutes)` 助手（供阶段 0.9 `compute_stage` 复用）；附样例 `data/seeds/seed_rose.tres`（crop_output_id="rose"）+ `data/crops/crop_rose.tres`（base ItemData, category=CROP）。**注册进 `_item_registry` 已随 2.3 在 main._ready 落地（`_register_farm_items` 扫描 data/seeds、data/crops）。**
- **步骤 0.6 — 分解机制泛化（任意可分解物品）**【✅ 已实现】
  - `decompose_recipe` 字段（Array[Dictionary]，每项 `{item_id, count}`，空 = 不可分解）已在 **0.1** 落到 `ItemData` 基类
  - 已落地：`GameManager.decompose(item_id) -> bool`（按配方产出进各自分类库存、原物扣 1；物品须已注册+有配方+持有≥1 才执行）+ `get_decomposables() -> Array`（列出背包内可分解物，供 2.4 分解子页）
  - 例：已置 `data/crops/crop_rose.tres` 配方 `玫瑰→红染料×1+纤维×2`；并新建 `data/materials/red_dye.tres`、`data/materials/fiber.tres`（base ItemData, category=MATERIAL）作产物样例。某布料→线×3 同理（待布料类物品落地）
  - 工坊「分解」子页 UI 留在 **2.4**（仅数据层，未碰 UI）
  - （原计划的 `CropData` 子类取消，配方直接挂在 ItemData 上，更通用）
- **步骤 0.7 — 新建 `BlueprintData` 资源类（定义类，非库存物）** `scripts/resources/blueprint_data.gd`【✅ 已实现】
  - **设计决定：`extends Resource`（不是 ItemData）**——蓝图是"制作配方/解锁定义"，不进 inventory，不应带 category/price/decompose_recipe/placeable_scene 等库存语义；解锁状态由 GameManager.unlocked_blueprints 跟踪（不存此处）。所有蓝图放 `res://data/blueprints/`，阶段 0.8 启动时统一加载。
  - 字段：`id`、`display_name`、`icon`、`description`、`unlock_inspiration`（灵感累计阈值，0=初始即解锁）、`required_materials`（Array[Dictionary]，每项 {item_id, count}）、`output_id`（产出 ItemData 的 id，CLOTHING/CRAFT）、`output_count`（默认1）
  - 助手：`materials_text(resolve)` 把 required_materials 渲染成"红染料×1 纤维×2"简介（resolve 传 GameManager.get_item 解析展示名，缺则回退 id）
  - 样例：`data/blueprints/blueprint_scarf.tres`（id="scarf_bp"，unlock_inspiration=20，需 red_dye×1+fiber×2 → 产出 scarf×1）+ `data/clothes/clothes_scarf.tres`（ClothesData, category=CLOTHING，id="scarf"，price=80）。与 0.6 形成完整链路：种玫瑰→分解得染料+纤维→达灵感阈值解锁→工坊制作出围巾。
  - 注意：required_materials 的 dict key 在 .tres 里必须加引号（0.6 已踩过裸标识符 Parse Error 的坑）。
- **步骤 0.8 — `GameManager` 增加蓝图解锁与灵感统计**【✅ 已实现】
  - 新增 `inspiration_total_earned`（单调递增，存档 `user://inspiration_total.cfg`）：在 `add_inspiration()` 对**所有正向灵感**（活动结算 + 订单奖励等）累加，负值（花费）不计 → 严格单调。若只想算"灵感活动"，把累加移入 `_finish_activity` 即可（一行之差）。
  - 新增 `unlocked_blueprints: Dictionary`（id→true）+ 信号 `blueprint_unlocked(id)`；方法 `_evaluate_blueprint_unlocks(emit)`（计划原名 `check_blueprint_unlocks`，实现为私有+emit 参数）在 `add_inspiration` 正数与启动初评时调用，达阈值解锁；存档 `user://blueprints.cfg`，已解锁者阈值日后调高也不回锁。
  - 启动时 `_load_blueprints()`：扫描 `res://data/blueprints/` 载入全部 `BlueprintData` 进 `_blueprint_registry`，恢复已解锁，再静默(`emit=false`)解锁已达阈值者。
  - 附查询访问器：`get_blueprint(id)` / `get_all_blueprints()`(按阈值升序) / `is_blueprint_unlocked(id)`（供 2.4 工坊）。
  - **制作动作 `craft()`（扣材料+产出）留在 2.4**（0.8 只做解锁与统计）。
- **步骤 0.9 — `GameManager` 增加农场槽状态（挂机友好，墙钟计时）**【✅ 已实现】
  - 新增 `farm_slots: Array`（长度 `FARM_SLOTS=6`，每槽 `{pot_id, seed_id, planted_unix, done}`）+ 信号 `farm_changed`
  - **方法（为对齐 2.3「先放盆、再放种子」两步 UI，计划的 `plant(slot,pot_id,seed_id)` 合一拆成两个独立方法）**：
    - `place_pot(slot, pot_id)`：空槽从 `inventory[PLACEABLE]` 取 1 个花盆放入（离开库存）
    - `plant(slot, seed_id)`：已有盆的槽从 `inventory[SEED]` 取 1 种种入（记 `planted_unix`=当前墙钟）
    - `compute_stage(slot)`：按 `SeedData.stage_at(已生长分钟)` 实时算 苗(0)/成长(1)/成熟(2)；空槽返回 -1
    - `is_slot_mature(slot)` / `get_growth_info(slot)`：`get_growth_info` 返回 `{stage, stage_name, progress(0..1), elapsed_sec, remaining_sec, total_sec}` 供 2.3 进度条+剩余 MM:SS
    - `harvest(slot)`：成熟→`add_item(crop_output_id)` 进 `inventory[CROP]` 并清空种子状态（花盆保留可重种）
    - `clear_slot(slot)`：花盆退回 `inventory[PLACEABLE]`，重置种子状态（回收花盆用）
    - `decompose(item_id)`：已在 0.6 实现，0.9 直接复用（不重造）
  - 存档：`user://farm.cfg`（每槽 pot_id/seed_id/planted_unix/done）；关游戏再开按墙钟续算，符合挂机游戏
  - **注意**：0.5 已建 `SeedData`（含 `stage_at`），种子/作物注册进 `_item_registry` 已随 2.3 在 `main._ready`（`_register_farm_items`）落地；花盆作为 ProductData 经 `product_pool` 注册。故 `plant/harvest` 运行时能正确解析 `SeedData`。

**阶段 0 验收**：购买/衣橱能写入 `inventory`；`get_by_category` 按类返回正确；Seed/Crop/Blueprint 资源可实例化；灵感达阈值能解锁蓝图；农场槽按墙钟续算；存档重载后数据在。

## 阶段 1 · 屏幕管理框架（ScreenManager + 底部 TabBar）

- **步骤 1.1 — 新建 `screen_manager.gd`**（建议作为 `main` 子节点或 autoload）
  - 持有 4 个屏幕引用（换装/种植/制作/仓库的 Control）
  - `show_screen(id)`：`hide` 其余、`show` 目标，记录 `current_screen`
  - 保证任意时刻仅 1 个活动屏，四屏互斥显隐（家园与种植/换装/工坊完全对等，切到其它屏时家园专属内容随之隐藏）；HUD 与关闭按钮上提至 Main 常驻
- **步骤 1.2 — 新建底部 `TabBar` 场景/节点** `scenes/tab_bar.tscn` + `scripts/tab_bar.gd`
  - 横栏 + 4 枚对等主 tab（家园/种植/换装/工坊），每枚是一个 `Control`（图标+文字）
  - 外加 1 枚**全局入口**（仓库），用分隔线与主 tab 区分，暖金基调（暗示「随时可达」）
  - 信号 `tab_selected(id)`
  - 默认高亮「家园」态（无覆盖屏活动）
- **步骤 1.3 — `main.gd` 接线**
  - 实例化 TabBar，挂在窗口底部
  - `tab_selected` → `screen_manager.show_screen(id)`
  - 家园桌面是 4 个对等屏之一（即 `$Panel`）：切到其它屏时其专属内容（角色/宠物/电话/摆放物/灵感按钮）随屏隐藏；HUD 与关闭按钮已上提 Main 常驻
- **步骤 1.4 — 确认窗口定位**【✅ 已实现】
  - 尺寸 `680×420`（`project.godot` 已设）、透明+无边框（`main.gd._configure_window()` 运行时 `borderless/transparent=true` + `per_pixel_transparency` 标志）均已满足
  - 右下角锁定：在 `_configure_window()` 末尾调 `_position_to_bottom_right()`，用 `DisplayServer.screen_get_usable_rect` + `window_set_position` 钉到当前屏幕右下角（排除任务栏）
- **步骤 1.5 — 左右箭头切换（贴在活动面板两侧）**【✅ 已实现】
  - 在活动面板**左右边缘**各放一个箭头按钮（‹ / ›），垂直居中，固定在面板边缘、不随内容滚动
  - 内容区左右各留约 28px **gutter**，避免被箭头遮挡（占位屏 `_make_placeholder_screen` 的 bg/label 已 inset 28px）
  - 点击 → `screen_manager.show_screen(prev/next)`；循环覆盖四个对等屏：家园→种植→换装→工坊→（回家园）；仓库为全局浮层不进循环
  - 复用 `tab_selected` 流程 → 对应底部招牌同样摆动（视觉反馈一致）
  - 可选：键盘 ← / → 同效（仅箭头可见时生效，`get_viewport().set_input_as_handled()` 防误触）
  - 注意：箭头**不放头部条**，用户明确要求贴在面板左右两侧
  - 实现：`scripts/arrow_nav.gd`（`class_name ArrowNav`）+ `scenes/arrow_nav.tscn`（根 `mouse_filter=IGNORE` 穿透点击，两枚按钮 STOP）；由 `main._setup_screens()` 实例化并 `add_child` 到 Main（绘制序在 Screens/TabBar 之上），`setup(screen_manager)` 接通 `screen_changed` → 仅在四个对等屏（home/farm/wardrobe/workshop）活动时显示；仓库浮层打开/关闭经 `set_overlay_blocked` 屏蔽/恢复（`_toggle_warehouse_global` 与 `_on_warehouse_exited` 两处兜底）

**阶段 1 验收**：点 4 个主 tab（含家园）能切换对应空屏；仓库全局浮层任意屏可开；面板两侧箭头能在四个对等屏（含家园）间循环；同时只一个切换屏活动。

> 家园（底层常驻场景）新增「服装展架」摆件，并改造现有顾客系统：从「点顾客消失给金币」改为「顾客购买展架上的衣服后消失」。

- **步骤 4.1 — `ItemData` 增加售价字段（售价取 `ItemData.price`，已确认）**
  - 新增 `price: int`（在展架售出时获得的金币；检查器可配；服装/制作物类生效，PLACEABLE 等不售）
  - 注：该字段已在阶段 0 基类落地（见 `scripts/resources/item_data.gd` 第 21 行），本步无需再建
- **步骤 4.2 — 新增服装展架摆件 `clothing_rack.tscn` + `clothing_rack.gd`**
  - 桌面可拖动摆件（沿用 phone 的 `_input + contains_point` + 拖动 / 位置存档模式）
  - 点击 → 打开「上架」面板（覆盖层，同 phone 订单中心模式）
  - 6 个槽位（常量 `RACK_SLOTS := 6`，暂定），每槽存 `{item_id}` 或空
- **步骤 4.3 — 上架面板 `rack_panel.gd/.tscn`**
  - 列出 `inventory[CLOTHING]` 中**可上架**的服装（排除已穿：`get_count(item_id) - equipped_count(item_id) > 0` 才可上架）；点一件 → 放入选中空槽（`GameManager.display_clothing(slot, item_id)`）
  - 同款可重复上架（允许多槽放同一 item_id）
  - 每个槽显示：服装图标 + **库存数量 = 拥有总数 − 穿戴数**（`get_count(item_id) - equipped_count(item_id)`）；只要仍拥有就**自动补充**——售出后若还有库存则槽持续显示，仅当拥有数（扣穿戴）归零才从展架消失
- **步骤 4.4 — `GameManager` 增加展架状态**
  - `clothing_rack: Array`（长度 RACK_SLOTS，元素 `item_id` 或 null，仅作「在售列表」不消耗库存）
  - 方法：`display_clothing(slot, item_id)`、`undisplay(slot)`、`get_rack_stock(item_id)`（= `get_count - equipped_count`）、`sell_from_rack(slot)`
  - `sell_from_rack(slot)`：校验槽非空 → `add_coins(price)` + `remove_item(item_id, 1)` → **若 `get_rack_stock(item_id) > 0` 槽保留（自动补充），否则清空该槽** → `emit rack_changed`
  - 存档：`user://rack.cfg`
- **步骤 4.5 — 改造顾客系统（现有 顾客 行为变更）**
  - 旧：顾客出现 → 点击 → 给金币 → 消失
  - 新（**随机选有货槽**，已确认）：顾客出现 → 随机走向一个 `get_rack_stock(item_id) > 0` 的槽 → 购买该槽服装 1 件（`sell_from_rack`）→ 付金币 → 消失；因自动补充，同款可继续被其他顾客买
  - 无货（展架空 / 全售罄）时顾客降级为「逛逛后离开」（不给金币，避免死锁）
- **步骤 4.6 — 桌面装饰花盆（限量）**
  - 花盆作为 PLACEABLE，除在种植屏 6 槽内作功能盆外，还可经现有 PlacementManager 摆到桌面作**装饰**，数量**限量**（常量 `DESKTOP_POT_LIMIT`，暂定如 3，后续再定）
  - 装饰盆为纯外观，不参与生长；与种植屏功能槽职责分离（两套互不影响）
- **步骤 4.7 — 数据联动**
  - 库存数量随 `inventory_changed` 实时刷新；售出后槽按库存自动补充/清空、仓库角标同步

**阶段 4 验收**：点击展架能上架服装、6 槽可重复同款、显示库存数量(=拥有−穿戴)；顾客随机购买后消失且金币增加；库存归零才从展架消失（否则自动补充）；关游戏再开展架与库存正确。
