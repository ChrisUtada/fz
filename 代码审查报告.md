## 二、P0（必须修）

### 1. `placement_manager.gd:63-66` — remove_placed 复活 bug（已验证）
`queue_free()` 不会立即把节点移出树，紧接着的 `save_placements()` 遍历 `get_children()` 时被删节点仍在，仍被写入存档 → 下次 `restore()` 复活（摆放物删不掉，越删越多）。

**修法 A（推荐）**：save 时跳过待删节点
```gdscript
for c in _container.get_children():
    if c is Placeable and c.data != null and not c.is_queued_for_deletion():
        # 写入存档
```

**修法 B**：先移出树再释放
```gdscript
remove_child(p)
p.queue_free()
save_placements()
```

## 三、P1（应修）

### 架构与耦合

**2. `game_manager.gd` 上帝单例（1383 行 / 9 职责）**
建议按领域拆为独立 autoload：`EconomyManager` / `ActivityManager` / `InventoryManager` / `WardrobeManager` / `FarmManager` / `RackManager`。
成本高但是最重要的长期债务；可渐进式拆，先抽 `InventoryManager` 作为样板。

**3. `game_manager.gd:704-729` craft / `678-697` decompose 批量操作频繁写盘 + 发信号（已验证）**
一次制作 N 个材料 = N 次 remove + 1 次 add，触发 **N+1 次 `_save_inventory()` 磁盘写 + N+1 次 `inventory_changed.emit()`** → UI 刷新 N+1 次。
建议加 `begin_batch() / commit_batch()` 模式，批量内只存一次、只 emit 一次。

**4. UI 直读 GM 内部字段（违反低耦合）**
- `farm_screen.gd:81` 读 `farm_slots[i]`
- `workshop_screen.gd:98` 读 `unlocked_blueprints.has()`
- `wardrobe.gd:92,203` 读 `unlocked_clothes.keys()` / `equipped.keys()`

建议补 getter 收口：`get_slot(i)` / `get_equipped_dict()` / `is_blueprint_unlocked(id)` / `get_unlocked_clothing_ids()` 等，UI 只通过接口访问。

**5. `game_manager.gd:684-697` decompose 弱类型（已验证）**
`decompose_recipe` 用 `Array[Dictionary]` + id 字符串，每次分解重建 `norm` 字典处理 String/StringName 键；而 `blueprint_data` 用强类型 `MaterialCost`。
统一改 `Array[MaterialCost]` 可删掉整个 workaround（约 20 行）。

### bug / 隐患

**6. `hud.gd:33-34` tween 未 kill（已验证）**
注释写「终止旧动画」，实际 `_tween = create_tween()` 没 kill 旧实例，快速货币变化时多条 tween 并行写 `_display_gold` 抖动。
```gdscript
if is_instance_valid(_tween):
    _tween.kill()
_tween = create_tween()
```

**7. `pet.gd:50,52` 初始化顺序定时炸弹（已验证）**
`_ready` 里 `_pick_direction()`（用默认 `edge_margin=50`）在 `_apply_visuals()`（覆盖为 `PetData.edge_margin`）之前；若某 PetData.edge_margin<50，定位后 `_check_bounds` 立即判出界 → 瞬灭。当前被 `pet_cat.tres=50` 掩盖。
修：`_ready` 里先 `_apply_visuals()` 再 `_pick_direction()`。

**8. `inspiration_panel.gd:299` 衰减系数与 GM 重复**
`maxf(0.5, 1-n*0.2)` 魔法数在 UI 和 `game_manager.gd:391` 双处维护。GM 暴露 `get_reward_factor()`，UI 只读。

**9. `phone.gd:113-124` `_process` 每帧轮询 `get_arrived().size()`**
改订 `arrived_changed` 信号驱动（与农场/订单其它屏的响应式范式对齐）。

**10. `product_catalog.gd:174` / `receipt_panel.gd:85` 忽略 `start_order` / `confirm_receipt` 返回值**
失败仍 `queue_free()` 关闭。检查返回值，失败给提示而非静默关闭。

**11. `inspiration_panel.gd:128` `start_activity` 返回值未校验**
若 GM 拒绝（已有活动进行）UI 仍 `_show_running()`，状态不一致。`start_activity` 应返回 bool 并检查。

**12. `clothes/align_tool.gd` 引用已删字段 `equip_offset`**
脚本解析失败，工具已损坏。删除或重写为共享画布预览工具。

**13. `arrow_nav.gd:30` `setup()` 无幂等保护**
重复调用重复 `connect(screen_changed)`。加 `_initialized` 守卫或先 `disconnect`。

**14. `focus_bar.gd:14` `@onready var _title` 无赋值表达式却手动 new（误导）+ `:124` `await 0.9s` 后未加 `is_instance_valid(self)` 守卫**
节点期间 `queue_free` 会报错。

## 四、P2（可选优化，合并列举）

### 重复代码
- `_format_time` 四处、`_make_icon` 三处、phone/clothing_rack 拖拽状态机、pet/customer `contains_point` 样板 → 各抽公共函数 / 基类。

### 硬编码
- `placement_manager.gd:51` 中心 `(340,240)` / `POT_ID="pot"`
- `product_catalog` / `receipt_panel` 的 `res://data/product_*.tres` 回退路径
→ 常量化 / `@export` 注入。

### 性能
- `placement_manager.gd:58` 每次 release 写盘 → 摆放物多时卡顿，节流（如 0.5s 合并）。
- `customer.gd` String 状态机（idle/shopping/...）应改 enum；`browsing` 离开也 `emit order_completed`，信号名误导。

### 类型 / 数据设计
- `item_data.gd:27` 注释「仅四类」与枚举 5 值（含 DECOR）不符。
- `clothes_data` 继承的 `icon` 与自有 `icon_texture` 语义重叠。
- `pet.gd` `@export` 与 PetData 双重声明冗余；`_position_at_edge` 的 top/bottom 分支死代码；pet 用 local position 判界 vs customer 用 global_position 不一致。
- `customer_data` / `pet_data` `region_rect` 默认 `Rect2()`（空）— 漏填则贴图不可见，默认应给安全值。

### 存档 / 其它
- `cfg.save()` 返回值多处未检查。
- 缺统一存档版本号 / 迁移框架（迁移逻辑散落）。
- `receipt_panel` 未监听 `arrived_changed`，开着时新到货不刷新。
- `inspiration_panel:232` 字符串跨层调 `Main._enter_focus_mode` → 改信号。
