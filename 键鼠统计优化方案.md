# 键鼠统计功能优化方案

> 范围：外出活动（OUTING 模式）的「真实键鼠输入计数 → 折算灵感」功能。
> 目标：①解决「不能马上开始监测」；②键鼠统一为单一计数；③评估是否可用更优/原生方法。
> 状态：**方案文档，供决策参考，尚未实施**。

---

## 一、现状

### 1.1 功能定位
- 外出活动期间，统计玩家在 PC 上的**真实键盘按键 + 鼠标点击次数**，按 `inspiration_per_action` 折算灵感。
- 关键约束：用户外出时在**别处**用电脑（不一定对着宠物窗口），所以必须是**全局后台**计数，不能只收宠物窗口内的输入。

### 1.2 现有实现链路
```
inspiration_panel「开始外出」
   └─▶ GameManager.start_activity(OUTING)
         └─▶ _start_companion()          # 含 await，但调用方未 await
               └─▶ OS.create_process(python.exe, input_listener.py)
                     └─▶ Python: 注册 message-only 窗口 + RegisterRawInputDevices
                           └─▶ 每 50ms 写 user://outing_counts.json
                                 └─▶ GameManager 轮询 _read_companion_counts()（0.1s 一次）
                                       └─▶ UI 显示「键 X / 鼠 Y 次」
```

### 1.3 涉及文件
| 文件 | 角色 |
|---|---|
| `tools/companion/input_listener.py` | Python 监听器（Raw Input + ctypes，373 行） |
| `scripts/game_manager.gd` | 启动/读取/折算（`_start_companion`/`_read_companion_counts`/`get_outing_counts`） |
| `scripts/inspiration_panel.gd` | UI 显示（`_update_outing_counts`） |
| `data/activity_outing.tres` | 外出活动数据 |

## 二、问题诊断（实测）

### 2.1 监听器本身是好的
直接命令行跑 `input_listener.py`：**<100ms** 即写出 `{"running":true,"keys":0,"mouse":0,"error":""}`，Raw Input 注册成功，无报错。Python `import ctypes` 冷启动实测 **~0.16s**。所以瓶颈不在 Python 脚本，在 Godot 侧的启动编排。

### 2.2 「不能马上开始」的 5 个叠加原因

| # | 原因 | 影响 | 代码位置 |
|---|---|---|---|
| ① | 每次外出 spawn 全新 Python 进程 | 冷启动 ~0.3–0.5s 才写出首个就绪 JSON | `game_manager.gd:270 OS.create_process` |
| ② | **`start_activity` 调 `_start_companion` 未 `await`** | `_start_companion` 含 `await`，跑到首个 await 即交还控制权；`activity_started` 在监听器未就绪时早发 → UI 当帧查 `get_outing_counts()` 拿到 `active=false`，先显示「收拾行装中…」 | `game_manager.gd:196` |
| ③ | 就绪判定靠 0.1s 轮询 | 再叠 ~0.1s | `game_manager.gd:284` |
| ④ | **导出版致命**：`companion_script_path=res://tools/...py` 导出后在 .pck 内、磁盘无此文件 → Python 找不到脚本 → 轮询 3s 超时 → **永远「统计未启动」** | 导出版完全不能用 | `game_manager.gd:99` |
| ⑤ | `companion_python_path` 硬编码本机路径 `C:/Users/chris/.../python.exe` | 换机器/分发即失效 | `game_manager.gd:98` |

> 若你测的是**导出版**，真凶是 ④（完全启动不了）；若是**编辑器内**，则是 ①②③ 叠加的 ~0.3–0.5s「收拾行装中…」延迟。

## 四、方案对比

### 4.1 为什么 Godot 原生 `Input`/`_input` 不可行
Godot 的输入系统只收**落在 Godot 窗口上**的输入。外出活动要的是用户在别处用电脑时的**全局后台**计数，原生 Input 满足不了。❌ 排除。

### 4.2 当前 Raw Input API 是否正确
**正确**。`RegisterRawInputDevices` + `RIDEV_INPUTSINK` + message-only 窗口，是 Bongo Cat 同款做法：事件驱动（逐次精确）、后台可收、不注入全局钩子（不卡键鼠、不受 UIPI 影响）。**问题不在 API，在「宿主是 Python 子进程」**（冷启动 + 文件 I/O + 不可移植）。

### 4.3 三方案对比

| 维度 | 方案A：GDExtension(C++) | 方案B：常驻 Python | 方案C：最小补丁 |
|---|---|---|---|
| 即时开始 | ✅ 进程内常驻，reset 即时 | ✅ 启动即常驻，reset 即时 | ⚠️ 仍有 ~0.3s 冷启动 |
| 统一计数 | ✅ 一起改 | ✅ 一起改 | ✅ 一起改 |
| 导出版可分发 | ✅ .dll 随包，用户零依赖 | ❌ 用户须装 Python | ❌ 同 B |
| 性能 | ✅ 进程内、无文件 I/O | ⚠️ 50ms JSON 读写 | ⚠️ 同 B |
| 改动量 | 大（新增 C++ 工程 + 编译） | 中（改启动生命周期） | 小（await + 统一） |
| 需装工具链 | ✅ 需一次性装 MSVC/MinGW | ❌ 不需要 | ❌ 不需要 |
| 长期维护 | ✅ 最佳 | ⚠️ 仍背 Python 包袱 | ❌ 治标不治本 |

### 6.5 C++ 类设计：`InputCounter`

**职责**：进程内常驻，启动即注册 Raw Input，单一原子计数，GDScript 直接调 `get_count()`。零冷启动、零文件 I/O。

**关键设计**：
- 继承 `RefCounted`，由 GameManager 持有一个实例（整局会话常驻）。
- `start()`：起一条 **worker 线程**，在该线程创建 message-only 窗口 + `RegisterRawInputDevices`（键盘 0x06、鼠标 0x02，`RIDEV_INPUTSINK`），跑 `GetMessageW` 消息泵。**必须在注册线程上泵消息**，故用独立线程，不阻塞 Godot 主线程。
- `WndProc` 收到 `WM_INPUT`：解析，**按键 down（非 break）或鼠标任意键 down → `count.fetch_add(1)`**（统一单一计数，满足需求②）。
- `count` 为 `std::atomic<int>`，`get_count()` 原子读，线程安全。
- `reset()`：`count.store(0)`（外出开始时调用，等价于「取基线」）。
- `stop()`：`PostThreadMessage` 投 `WM_QUIT` 给 worker 线程并 `join`（进程退出时调用）。
- 开关键（CapsLock/NumLock/ScrollLock）跳过，与现有逻辑一致。

**对外接口（GDScript 可见）**：
| 方法 | 返回 | 说明 |
|---|---|---|
| `start()` | `bool` | 启动 worker 线程并注册 Raw Input；成功返回 true |
| `stop()` | `void` | 停止并清理 |
| `is_active()` | `bool` | 是否在统计中 |
| `get_count()` | `int` | 当前累计次数（统一键+鼠） |
| `reset()` | `void` | 计数归零 |

**伪代码骨架**：
```cpp
class InputCounter : public RefCounted {
    GDCLASS(InputCounter, RefCounted);
    std::atomic<int> count{0};
    std::atomic<bool> active{false};
    std::thread worker;
    HWND hwnd = nullptr;
    static LRESULT CALLBACK WndProc(HWND, UINT, WPARAM, LPARAM);
protected:
    static void _bind_methods();
public:
    bool start();           // spawn thread -> _thread_run
    void stop();
    int  get_count() const { return count.load(); }
    void reset()           { count.store(0); }
    bool is_active() const { return active.load(); }
    void _thread_run();    // RegisterClass + CreateWindow(MSG-only) + RegisterRawInputDevices + GetMessage loop
};
// _bind_methods: register start/stop/get_count/reset/is_active
```

### 6.7 GameManager 改造（要点）
- **删除**：`companion_python_path`、`companion_script_path`、`_companion_pid`、`_outing_json_path`、`_outing_ok`、`_outing_baseline`、`_start_companion`、`_read_companion_counts`、`_read_outing_error`、`get_outing_error`、`_kill_companion`。
- **新增**：`var _input_counter = InputCounter.new()`，`_ready()` 末尾 `_input_counter.start()`（整局常驻，一次性，无冷启动）。
- `start_activity(OUTING)`：`_input_counter.reset()` —— **即时**，无 spawn、无 await、无延迟。
- `get_outing_counts()`：`return {"total": _input_counter.get_count(), "active": _input_counter.is_active()}`。
- `finish_outing()` / `cancel_activity()`：不再 kill 进程；计数器继续常驻，下次外出再 reset。
- `tree_exiting`：`_input_counter.stop()`（可选；进程退出 OS 自动回收）。

### 6.8 inspiration_panel UI 改造（统一计数，需求②）
- `_outing_counts.text = "已输入 %d 次" % c.get("total", 0)`（删去「键 X / 鼠 Y」分列）。
- 删除「收拾行装中…」等待分支（reset 即时、active 立即为 true，无延迟窗口）。
- `_reward_hint`：`"统计中 · 已输入 %d 次 · 预计 +%d 灵感"`。

### 6.9 导出 / 分发
- 导出预设里把 `addons/fz_input_counter/bin/*.dll` 纳入打包（.gdextension 自动随）。
- **不再需要** Python、`input_listener.py`、`outing_counts.json`，可从工程与 `.gitignore` 清理。
- 用户端零依赖，Steam 直接可分发。

## 七、备选方案设计（常驻 Python，方案B）

> 不装工具链时的过渡方案。复用 100% 现有 Raw Input 逻辑，只改「生命周期」。

### 7.1 改造点
- 游戏启动（`GameManager._ready`）即 `OS.create_process` 拉起监听器，**整局常驻**（不再每次外出 spawn）。
- `start_activity(OUTING)`：读一次当前 `keys+mouse` 作基线（或调 Python 暴露的 reset）——**即时**。
- `get_outing_counts()`：`total = (cur.keys + cur.mouse) - (baseline.keys + baseline.mouse)`。
- `finish_outing`/`cancel`：不 kill 进程，下次外出重取基线。
- `tree_exiting`：kill 进程。

### 7.2 统一计数（需求②）
- Python 侧：`count = keys + mouse`（或直接维护单一 `count`，任何 down 都 +1）。
- UI 侧：`"已输入 %d 次"`。

### 7.3 优缺点
- ✅ 改动小、复用现成代码、解决「不能马上开始」、统一计数。
- ❌ 导出版仍不可用（`res://` 脚本路径 + Python 依赖），需后续补嵌入式 Python 或转方案A。
- ❌ 仍背 50ms JSON 文件 I/O。

## 八、统一计数（需求②）落地说明

无论选哪个方案，统一计数都一起做：
- **数据层**：单一 `count`（任何键盘按下或鼠标任意键按下 → +1）。
- **UI 层**：`inspiration_panel` 显示「已输入 N 次」，删除「键 X / 鼠 Y」分列。
- **折算**：`灵感 = ceil(count * inspiration_per_action)`，逻辑不变。

## 九、风险与注意事项

| 项 | 说明 | 应对 |
|---|---|---|
| 杀毒软件 | Raw Input 是官方 API、不注入钩子，一般不拦；但部分杀软对子进程/未签名 dll 敏感 | 方案A 的 .dll 建议长期做代码签名；方案B 的 Python 进程可能被拦 |
| UIPI / 管理员权限 | `RIDEV_INPUTSINK` 后台可收，不受前台限制；但目标为提权窗口时仍收不到 | 文档注明：用户以管理员运行其它程序时，非提权宠物收不到该程序输入（与 Bongo Cat 同限制） |
| 跨平台 | 仅 Windows（项目本身仅 Win） | C++ 侧 `#ifdef _WIN32` 守卫；非 Win 退化为 `is_active()=false` |
| 隐私 | 只统计「次数」，不记录按键内容 | 已有声明，保持；杀软友好 |
| 导出打包 | 方案A 需把 .dll 纳入导出预设 | 在 `export_presets.cfg` 确认 include filters |
| 线程安全（方案A） | worker 线程写、主线程读 | 用 `std::atomic` 计数，无需锁 |

## 十、验收标准

- [ ] 外出开始**即时**进入计数态（无「收拾行装中…」、无可见延迟）

## 附：决策摘要

- **核心诊断**：监听器本身 OK，瓶颈在 Godot 启动编排（未 await + 每次冷启动 + 导出 `res://` 路径失效 + 硬编码 Python 路径）。
- **本机现状**：无 C++ 工具链，方案A 需一次性安装。
- **推荐**：长期走方案A（原生，可分发）；短期可选方案B 过渡。
- **统一计数**：两方案都一起做。
